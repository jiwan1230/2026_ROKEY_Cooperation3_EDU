# 동일 크기 박스 적층 검출 — 시도/시행착오 기록

이 문서는 "위/아래 두 박스의 크기와 모양이 같을 수도 있다"는 새 요구사항을 풀면서
시도한 방법들, 왜 그 방법을 골랐는지/버렸는지, 최종적으로 뭘 썼는지를 시간순으로
기록한다. 사용자가 다음 대화에서도 진행 상황을 알 수 있도록, 코드만 보고는 알 수 없는
"왜"를 남기는 게 목적이다.

## 0. 문제 정의

기존 파이프라인(`box_geometry.py`)은 "위 박스는 항상 아래 박스보다 작다"는 물리적
전제를 여러 곳에 깔고 있었다(`SUPPORT_MIN_AREA_RATIO` 타이브레이크 등). 이 전제가
깨지면(위아래 박스 크기가 같으면) 아래 박스의 노출된 테두리(exposed rim)가:
- 아예 없거나(완전히 같은 XY에 정렬된 경우 - 이 경우는 시각적으로 원천적으로 구분
  불가능하다. 카메라가 몇 개든, 어느 각도든 위 박스가 아래 박스를 완전히 가리면
  "박스 1개(더 큰 높이)"와 "박스 2개(작은 높이 합)"를 구분할 정보 자체가 없다 -
  이건 알고리즘 문제가 아니라 정보 이론적 한계다.
- 있어도 매우 얇다(몇 mm~수 cm 수준) — 기존 파이프라인의 `make_candidate()`가
  요구하는 `MIN_BOX_SIDE_M=0.04`(4cm) 최소 변 길이 조건을 통과하기엔 너무 얇아서,
  RANSAC/DBSCAN이 아무리 좋은 데이터를 받아도 "독립적인 박스 후보"로 잡히지 않는다.

즉 기존 방식("각 평면을 독립적인 사각형 후보로 찾고, 크기 필터를 통과시킨다")은
구조적으로 이 케이스를 풀 수 없다. 완전히 다른 전략이 필요하다.

## 1. 시도한 접근들

### 1-A. (기각) 2D 그리드 템플릿 매칭
윗박스(이미 정확히 검출됨)의 사각형을 (dx, dy) 오프셋 그리드 전체에 대해 스캔하면서
아래쪽 점들과의 일치도를 점수로 매기는 방식. 구현은 확실하지만:
- 탐색 격자 해상도/범위를 정해야 해서 파라미터가 늘어난다
- 얇은 테두리 하나만 있어도 되는데 굳이 2D 전체를 스캔할 필요가 없다
- 회전까지 고려하면 탐색 공간이 3차원(dx,dy,dθ)으로 커진다

### 1-B. (채택) 변(edge)별 돌출량 직접 측정
윗박스 사각형의 로컬 좌표계(rect_u, rect_v)로 전체 포인트를 투영한 뒤, "윗박스
사각형 경계 바로 바깥쪽 + 윗박스 윗면보다 낮은 위치"에 있는 점만 추린다. 이 점들이
윗박스의 어느 변(±u, ±v) 쪽에 있는지에 따라 그 방향으로의 돌출량(protrusion)을
직접 계산한다 - 아래 박스가 dx만큼 +u 방향으로 밀려 있으면 +u쪽 변에서만 dx만큼
돌출된 점들이 관측되고 반대편(-u)에는 안 보인다(위 박스에 가려짐)는 기하학적 사실을
그대로 이용한다.

이 방식을 택한 이유:
- 얇은 테두리 하나(4cm 미만이어도)만 있으면 오프셋을 추정할 수 있다 - 최소 변
  길이 조건이 없다
- 회전이 같다고 가정하면(대부분의 실제 적층 상황) 2D 격자 탐색 없이 각 축별로
  독립적으로, 닫힌 형태로 오프셋을 계산할 수 있다(회전까지 다르면 이 가정이
  깨지는데, 우선은 "같은 방향으로 쌓인" 케이스부터 푼다 - 회전까지 다른 케이스는
  범위 밖으로 명시적으로 남겨둔다)
- 기존 파이프라인의 지지면 탐색(select_support_candidate/detect_floor_boundary)과
  같은 "점이 경계 근처에 있는지" 패턴이라 구현/디버깅 방식이 일관된다

## 2. 언제 이 탐색을 돌리는가

모든 박스에 대해 항상 돌리면 계산 낭비 + 오탐 위험이 있어서, "윗박스가 최종적으로
바닥(floor)을 지지면으로 찾은 경우"에만 이 탐색을 추가로 시도한다 - 숨겨진 동일
크기 박스가 있다면 정확히 이 실패 모드(중간 박스를 건너뛰고 바닥까지 내려가 버림)로
나타나기 때문이다. 이미 다른 박스 후보를 지지면으로 정상적으로 찾은 경우
(support_type=box_top)는 이미 서로 다른 크기의 적층으로 잘 풀린 것이므로 건드리지
않는다.

## 3. 시행착오 로그

### 3-1. 변별 돌출량을 중앙값으로 재서 실제 오프셋의 절반으로 과소추정됨
합성 데이터(numpy로 만든 바닥+적층 쌍)로 처음 테스트했을 때, 실제 오프셋
(예: 0.04m)이 추정값 0.02m로 정확히 절반씩 나왔다. 원인: 노출된 테두리 위의
점들은 top의 경계선(half_u)부터 아래 박스의 실제 바깥쪽 모서리까지 폭 전체에
걸쳐 고르게 분포하는데, 그 중앙값을 "돌출량"으로 썼다 - 중앙값은 당연히 그
구간의 중간(=진짜 돌출량의 절반)을 가리킨다. **수정**: 중앙값 대신 상위
백분위수(90번째)를 쓴다 - 진짜 모서리는 분포의 가장 바깥쪽에 있다. 수정 후
오프셋 2mm~60mm 범위에서 오차 0.2~6.7mm로 줄어듦(실측표는 3-3절 참고).

### 3-2. 완전히 안 보이는 경우(오프셋≈0) 바닥을 "숨겨진 박스"로 오탐
오프셋을 정확히 0으로 준 테스트 케이스(원천적으로 시각적 구분이 불가능한
케이스)에서, 알고리즘이 "찾았다"며 실제로는 훨씬 아래의 바닥을 아래 박스로
잘못 보고했다(z 오차 99.9mm). 원인: 다중 시점 스캔이 원래 완전히 가려져야 할
바닥 근처까지 비스듬한 각도로 일부 봐버려서, 그 바닥 점들이 "top 경계 바깥쪽
탐색 밴드" 안에 들어와 "가장 얕은 높이 군집"으로 오인됐다. **수정**: 호출부가
이미 알고 있는 "더 먼 지지면까지의 깊이"(`known_far_support_depth_m`)를
넘겨받아서, 새로 찾은 높이가 그 바닥과 사실상 같으면(최소 박스 높이만큼도
차이 안 나면) 기각하도록 방어 로직 추가. 수정 후 오프셋=0 케이스는 정확히
"못 찾음"(None)을 반환 - 이건 실패가 아니라 "정보 이론적으로 답이 없는
질문에 모른다고 정직하게 답한 것"이다(0절 참고).

### 3-3. 최종 합성 데이터 검증 결과
`box_geometry.find_hidden_stacked_box()` 단독 테스트(바닥+적층 쌍만):

| 진짜 오프셋(du,dv) | 추정 XY 오차 | 추정 Z 오차 |
|---|---|---|
| (0.02, 0.01) | 1.4mm | 0.0mm |
| (0.04, 0.0) | 2.5mm | 0.0mm |
| (0.0, 0.03) | 1.6mm | 0.0mm |
| (0.06, 0.05) | 6.7mm | 0.0mm |
| (0.01, -0.01) | 1.1mm | 0.3mm |
| (0.005, 0.0) | 0.7mm | 0.1mm |
| (0.002, 0.0) | 0.2mm | 0.0mm |
| (0.0, 0.0) | "못 찾음"(정답) | - |

`multiview_scan.detect_boxes_in_base_frame()` 전체 파이프라인 테스트(바닥 +
적층 쌍(오프셋 0.025,-0.015) + 무관한 독립 박스 1개, 총 23312점):
- 적층 쌍: 위 박스가 원래(버그 상태) "support=floor, 잘못된 높이(두 박스 합)"로
  나오던 게, 수정 후 "support=box_top(새로 찾은 아래 박스), 높이 0.07m(실제
  top만의 높이 0.08m - margin 1cm 차이는 의도된 것)"로 정확히 보정됨. 새로
  찾은 아래 박스도 별도 박스로 정확히 복원됨(중심 오차 <1cm).
- 무관한 독립 박스: 전혀 영향 없음(오탐/오검출 없음) - 이 탐색은 support_type
  =="floor"인 박스에만 시도하므로 이미 정상 검출된 박스는 안 건드린다.

### 3-4. 실제 Isaac Sim 데이터 첫 적용 - `_group_by_location`이 XY만 보던 버그

`35.crate_scan_setup.py`의 `TABLE_BOXES`를 Small=Large(둘 다 0.23x0.16x0.14)로
바꾸고 실제로 스캔한 point cloud(`scan_cache/merged_table_scan.npy`, 9시점
698,158점)에 위 로직을 처음 돌려보니, Small이 최종 결과에 아예 안 나타났다.

원인: `multiview_scan._group_by_location()`이 시도별(trial) 검출 결과를
"윗면 사각형 중심의 XY 거리"로만 묶고 있었다 - Small을 Large 바로 위(같은 XY,
다른 Z)에 스택시켰기 때문에, Small의 모든 시도 인스턴스가 Large와 같은 그룹으로
묶여버렸고, 그 그룹 안에서는 fill_ratio가 항상 더 높은 Large만 채택됐다(각
그룹에서 fill_ratio 최댓값 하나만 선택하는 로직이므로). `35.crate_scan_setup.py`의
`table_real_boxes` dedup에도 있었던 것과 같은 종류의 실수다.

**수정**: `_group_by_location()`에 Z 거리 조건(`DEDUP_OVERLAP_Z_TOLERANCE_M`
재사용)을 AND로 추가 - XY도 가깝고 Z도 가까워야 같은 그룹으로 묶는다.

### 3-5. 테이블 전체 표면이 지지면으로 오인됨 (Large의 높이가 2배로 나옴)

위 수정 후 Small은 자기 그룹으로 잘 분리됐지만, 계산된 높이가 0.268m(참값의
약 2배)로 나왔다. 원인 추적: `select_support_candidate()`가 Large의 노출된
테두리(너무 얇아서 애초에 독립 후보로 안 잡힘 - 0절 참고) 대신, 우연히 "유효한
평면 후보"로 검출된 테이블 표면 전체(footprint 약 0.8x0.6m)를 Small의 지지면
("box_top" 타입)으로 잘못 채택하고 있었다 - 테이블은 평평하고 넓어서 사각형
채움비 검사를 쉽게 통과한다.

**수정**: `select_support_candidate()`에 `max_support_area_ratio` 파라미터 추가
- top 자신의 면적보다 이 배수(6.0으로 설정) 이상 큰 후보는 지지면에서 제외한다.

### 3-6. `support_type` 라벨과 무관하게 탐색해야 했음 + 저신뢰 후보 오탐 배가

3-5 수정 후에도 `_split_hidden_same_size_stacks()`가 처음엔 `support_type=="floor"`
인 박스만 검사하도록 짜여 있었는데, 3-5의 버그 때문에 실제로는 `support_type`이
`"box_top"`(테이블을 박스로 착각)으로 나오고 있어서 탐색 자체가 시작되지 않았다.
**수정**: support_type과 무관하게 항상 시도하도록 변경.

그런데 이렇게 바꾸자, fill_ratio=0.716(12번 중 6번만 나타난 저신뢰 후보)에서
탐색이 실행돼 존재하지 않는 오탐 박스가 하나 더 생겼다. **수정**:
`MIN_TRUSTED_FILL_RATIO_FOR_HIDDEN_SEARCH=0.85` 게이트 추가 - 이미 충분히
신뢰할 만한 top 후보에만 탐색을 적용.

### 3-7. (폐기) 변별 돌출량 히스토그램 하나로 "있는지 + 깊이"를 동시에 찾는 방식

`find_hidden_stacked_box()`의 원래 설계는, top 경계 바로 바깥쪽 점들을 깊이별
히스토그램으로 쌓고 "밀도 있는 첫 bin"을 아래 박스의 깊이로 채택하는 방식이었다.
실측 확인 결과 이 방식은 두 방향으로 다 실패했다:

- **못 찾아야 할 곳에서 찾음**: 독립적으로(안 쌓인 채) 놓인 Medium에서도 이
  히스토그램이 "숨겨진 박스"를 계속 오탐했다 - 원인은 Medium 자신의 옆면
  (수직으로 이어지는 면)이 여러 깊이 bin에 걸쳐 고르게 점을 남기기 때문
  (실측: bin당 약 244~248점, 3~4개 연속 bin). **수정 시도**: "피크 다음 bin의
  밀도가 뚜렷이 떨어지는지"(`WALL_VS_SURFACE_DROPOFF_RATIO=0.5`) 조건 추가,
  `MIN_MEANINGFUL_PROTRUSION_M`을 1cm→2cm로 상향.
- **찾아야 할 곳에서 못 찾음**: 위 조건을 추가하고 나니, 이번엔 진짜 적층
  케이스(Small/Large)에서도 매 bin마다 "다음 bin이 안 떨어짐"만 반복하다 결국
  못 찾는 반대쪽 실패가 나타났다(실측: bin 1~6 모두 옆면으로 오인되고 기각).
  bin 경계를 관측 샘플 상대값이 아니라 절대(h=0 기준) 위치로 고정하는 등
  여러 튜닝을 시도했지만, "옆면과 진짜 표면을 히스토그램 하나로 안정적으로
  구분"하는 문제 자체가 두 요구사항(민감해야 오탐 안 놓침 vs 둔감해야 오검출
  안 함)이 서로 충돌해서 근본적으로 불안정했다.

이 접근은 결국 폐기했다(아래 3-8절 참고). 이 절은 "왜 히스토그램 방식이 안 됐는지"
기록으로 남긴다.

### 3-8. (채택) `detect_floor_boundary()` 재귀 재탐색 - 히스토그램을 완전히 대체

`find_hidden_stacked_box()`를 고치는 대신, 완전히 다른 각도로 접근했다:
`detect_floor_boundary()`가 이미 "top 바로 아래에서 가장 가까운 평면"을 찾도록
설계돼 있다는 점(정렬 키 `(median_distance, -area, -hit_count)`)에 주목했다 -
숨겨진 동일 크기 박스가 있으면, 그 박스 자신의 노출된 윗면이 정확히 "가장
가까운 평면"이 되어 우연히(원래 의도한 동작은 아니지만) 찾아진다.

실측 확인(Small/Large 실제 데이터, top의 detect_floor_boundary를 15회 반복):
매번 0.136~0.151m 범위(참값 0.14m와 거의 일치)로 안정적으로 Large의 rim을
찾아냈다 - 반면 원래 코드는 이 결과를 그냥 "바닥"으로만 쓰고 끝냈다(그 지지면
자체가 사실 또 다른 박스의 윗면일 수 있다는 걸 검증 안 함).

**새 알고리즘**(`box_geometry.find_stacked_layers()`): `detect_floor_boundary()`가
찾아준 지지면을, top과 같은 footprint를 가진 "새 top 후보"로 간주하고 그 자리에서
`detect_floor_boundary()`를 다시 호출한다 - 거기서도 유효한 지지면을 찾으면
방금 찾은 건 바닥이 아니라 "또 다른 박스"였다는 뜻이다. 진짜 바닥이라면 스캔
크롭 범위(`TABLE_TOP_Z - 0.05m` 아래는 point cloud에 없음) 밖이라 더 아래에서는
찾을 수 없다 - 실측 확인: Medium(실제로 안 쌓인 독립 박스)에서 20회 중 12회
정확히 "더 없음"으로 멈췄다(나머지는 아래 3-8-1 노이즈 참고). 이 반복 재탐색을
여러 번(9회) 돌려서 "몇 겹까지 내려가는가"를 다수결로 정하고, 채택된 겹수만큼만
각 단계 깊이를 중앙값으로 취한다.

이 접근이 히스토그램 방식보다 훨씬 안정적인 이유: 이미 실전에서 검증되어 쓰이고
있던 `detect_floor_boundary()`(RANSAC 평면 검출 + ray-cast 경계 확인)를 그대로
재사용해서, 새로 만든 코드는 "그 결과를 얼마나 신뢰하고 얼마나 반복할지"를
결정하는 다수결 로직뿐이다 - 새로 튜닝해야 하는 임계값(히스토그램 bin 폭,
dropoff 비율, 최소 돌출량 등)이 훨씬 적다.

#### 3-8-1. Medium에서 20회 중 2회 크롭 경계 노이즈를 "층"으로 오인

Medium(독립 박스)에 대해 재귀 재탐색을 20회 반복했더니, 18회는 정확히 1겹(바닥)
에서 멈췄지만 2회는 크롭 경계(`TABLE_TOP_Z - 0.05m`) 바로 안쪽의 노이즈를 2번째
층으로 오인했다 - 다만 그 "층 두께"가 3.6~3.7cm로 비정상적으로 얇았다(실제 박스
중 가장 얇은 것도 0.12m). **수정**: `MIN_PLAUSIBLE_LAYER_HEIGHT_M=0.06`(실제
카탈로그 최소 높이보다 넉넉히 낮은 하한) 미달인 층은 무시. 여기에 더해 여러 번
반복 + 다수결(`STACK_DESCENT_MIN_AGREE_FRACTION=0.5`) 채택 방식 자체가 10%
확률로만 나오는 이런 노이즈를 자연히 걸러낸다(다수결 기준을 못 채움).

#### 3-8-2. 2단계(진짜 바닥) 재탐색이 먼 곳의 다른 물체와 평면을 합침

Large의 rim에서 진짜 테이블까지 2단계로 내려가는 재탐색에서, 참값(2층 합
~0.28m)보다 3~4cm 짧게 나오는 현상을 발견했다. 원인: `detect_floor_boundary()`는
RANSAC을 **전체 scene**에 대해 돌리기 때문에, 진짜 테이블 평면이 "우연히 비슷한
깊이에 있는 완전히 다른 물체"(Medium의 윗면 - Large의 rim에서 테이블까지의
깊이와 Medium 자신의 높이가 우연히 비슷했다)와 하나의 평면으로 잘못 합쳐질 수
있다. 합쳐진 평면의 피팅 결과(중심/기울기)가 진짜 테이블보다 얕은 쪽으로
치우쳐서 깊이가 짧게 나온 것으로 추정.

**수정**: `_crop_local_region()` 추가 - top의 footprint를 margin(`STACK_DESCENT_
LOCAL_CROP_MARGIN_M=0.12m`)만큼만 확장한 국소 영역으로 미리 잘라낸 뒤
`detect_floor_boundary()`를 돌린다. 다만 **1단계(원래 top 바로 아래)에는 이
크롭을 적용하면 안 된다** - 시행착오로 확인: 둘 다 국소 영역으로 하면 1단계
자체가 사라진다(Large의 노출된 rim은 그 자체로 점이 너무 적어서, 국소 영역만
으로는 `detect_floor_boundary()`의 최소 점 개수/면적 기준을 못 채우고 무시됨 -
반대로 전체 scene에서는 "우연히 비슷한 높이의 먼 평면과 합쳐진 덕에" 오히려
기준을 통과해서 찾아짐). 그래서 1단계는 전체 scene, 2단계부터는 국소 영역으로
나눠 적용했다. 수정 후 실측(8회 반복): 총 깊이가 0.2793~0.2795m로 매우 안정적
(참값 0.28m와 거의 일치, 1단계 자체도 0.14m 근처로 안정).

### 3-9. XY 오프셋(du, dv) 정밀도의 한계 (⚠️ 이 절의 결론은 3-13절에서 뒤집힘 - 아래 3-13절 먼저 참고)

높이/개수(존재 여부)는 위 수정들로 정확해졌지만, 숨겨진 박스의 XY 오프셋
추정치는 설계 스폰 오프셋(월드 프레임 기준 (0.03, -0.02)m, 크기 3.6cm)과
직접 비교하면 다소 어긋난다(측정치 크기 약 5.16cm, 방향 검증 시 `m0609_base_link`
프레임이 world 프레임과 회전돼 있어(테이블 자체의 검출된 변 방향이 base_link
+X축과 정확히 90도 어긋남을 확인) 성분별 직접 비교는 불가능했고, 크기만 비교
가능했다).

원인 후보: (1) 낙하 물리 시뮬레이션 특성상 스폰 위치와 실제 정착 위치가
정확히 일치하지 않을 수 있음(범핑/미끄러짐), (2) 완벽한 평행이동이 아니라
미세한 회전이 섞여 정착했을 가능성(디버그 로그에서 `+u`가 노이즈 임계값
바로 아래(19.2mm<20mm)에서 배제되고 `-u, +v, -v` 세 변 모두에서 증거가
잡힌 것이 그 방증 - 순수 평행이동이라면 축당 한쪽 변에서만 증거가 나와야
한다), (3) 3.6cm라는 오프셋 자체가 이 알고리즘이 안정적으로 잴 수 있는
범위의 하한에 가까움(0절에서 이미 밝힌 "오프셋이 0에 가까울수록 정보이론적
한계에 가까워진다"는 성질의 연장선).

**시도한 대안**: 양쪽 변이 모두 기준을 통과할 때 평균 대신 "점이 더 많은 쪽만
채택"하는 방식을 시도했다 - 회전이 섞이면 반대쪽 변의 신호가 노이즈라는
가설이었다. 실측 결과 오히려 참값에서 더 멀어졌다(크기 5.16cm -> 6.88cm) -
가설이 틀렸거나, 두 변 모두 부분적으로 진짜 신호를 담고 있어서 평균 쪽이 더
나은 것으로 판단, 원래(평균) 방식으로 되돌렸다.

**현재 상태**: 박스의 존재 여부/개수/높이/footprint 크기는 반복 검증 결과
매우 안정적이고 정확하다(footprint 0.230x0.159m로 참값 0.23x0.16m과 거의
일치, 높이도 참값 대비 수 mm~1cm 이내). XY 오프셋은 방향/크기 모두 참고용
정밀도로 남겨둔다 - 이 데모 시나리오의 실제 오프셋(3.6cm)이 알고리즘이 잡음과
신뢰성 있게 구분할 수 있는 하한에 가까운, 상대적으로 어려운 케이스이기 때문이다.
더 큰 오프셋(5cm 이상)에서는 3-3절 합성 데이터 검증에서 이미 mm 단위 정확도를
확인했다.

### 3-13. XY 오프셋 부호가 실제로 틀려 있었음 (사용자 피드백으로 발견)

3-9절 작성 시점에는 "오프셋 정밀도의 한계"로만 기록했는데, 사용자가 최종 PLY를
직접 열어보고 "두 박스가 완벽하게 겹쳐지지 않았다", "상당 부분 겹친 것이 많다"고
지적했다. 재점검 결과 - 단순 정밀도 문제가 아니라 **방향(부호) 자체가 틀린
진짜 버그**였다.

검증 방법: (1) `m0609_base_link`이 world 대비 정확히 90도 회전돼 있다는 것을
테이블 자신의 검출된 변 방향(`table_top` 후보의 rect_u/rect_v가 base_link
+X축과 각각 -90도/0도)으로 직접 확인했다. (2) 이 회전을 이용해 `35.crate_scan_setup.py`
의 스폰 오프셋(world 기준 Small-Large = (0.03,-0.02))을 base_link 프레임으로
정확히 변환해 참값을 얻었다. (3) 원본 point cloud를 raw XY 산점도(색=높이)로
직접 그려서, Small 사각형 바로 위(+Y)쪽에 실제로 노출된 얇은 띠(Large의 rim)가
있는 걸 육안으로 확인했다 - 이게 유일하게 진짜 신호였다.

기존 코드(3-9절에서 "평균이 더 낫다"고 결론 내렸던 것)는 이 경우 `du`(u축)에서
`+u`(99점, 넓게 흩어짐)와 `-u`(333점, 상대적으로 조밀함)를 평균 내고, `dv`(v축)
에서도 `+v`(39점)와 `-v`(441점)를 평균 냈다 - 그런데 실제로는 점이 많은 쪽
(`-u`, `-v`)만 진짜 신호였고 점이 적은 쪽(`+u`, `+v`)은 노이즈였다. 평균을
내면 이 노이즈가 부호까지 뒤집을 만큼 신호를 오염시켰다(`dv`가 참값과 반대
부호로 나옴 - 시각화에서 확인한 "위쪽에만 노출된 rim"과 정반대 방향으로
사각형이 그려짐).

**수정**: "점이 더 많은 쪽만 채택"으로 재변경(3-9절에서 시도했다가 기각했던
바로 그 방식). 이번엔 시각화 + 정확한 좌표 변환으로 검증하고 나서 바꿨다 -
3-9절에서 이 방식을 기각한 근거였던 "world 오프셋과 직접 비교했더니 더
나빠짐"은, 애초에 base_link 회전을 고려하지 않고 잘못 비교한 결과였다(틀린
잣대로 내린 틀린 결론). 수정 후 8회 반복 검증: 오프셋 부호가 매번 일관되게
나오고(양쪽 축 모두), 시각화로 그려도 실제 노출된 rim 위치와 정확히 겹친다.
크기는 여전히 참값(3.6cm)보다 크게(5.2~6.3cm) 나오는데, 이는 남은 정밀도
한계로 이해한다(3-9절 사유 - 오프셋 자체가 작아 노이즈 대비 신호비가 낮은
케이스) - 그래도 방향은 이제 항상 맞다.

**교훈**: 참값과 비교할 때 좌표계 변환(여기서는 base_link의 90도 회전)을
빠뜨리면, 올바른 수정을 "더 나쁘다"고 잘못 기각할 수 있다. 다음에 비슷한
정밀도 이슈를 만나면, 숫자만 비교하지 말고 반드시 raw point cloud를
시각화해서 눈으로 직접 확인할 것.

### 3-14. 간헐적 유령 후보 (fill_ratio≈0.50)

8회 반복 검증 중 2회, `fill_ratio` 정확히 0.50~0.51인 후보가 3~4개 시도에서
일관되게 나타나 최소 등장 횟수 기준을 통과하고 유령 4번째 박스로 남았다(폭
필터는 통과할 만큼 충분히 넓었음). 진짜 박스 윗면은 항상 fill_ratio 0.9 이상
이었다. **수정**: `MIN_FINAL_FILL_RATIO=0.75` 필터를 최종 선택 단계에 추가.

### 3-15. 오프셋을 넓혀서(3.6cm -> 7.2cm) 재검증 - 최종 확정

3-13절 수정(방향 버그 수정)으로 부호는 항상 맞았지만, 사용자가 재생성한 PLY를
보고도 "여전히 겹치는 문제"를 지적했다. 재확인 결과: 3.6cm라는 원래 오프셋
자체가 박스 크기(~20cm)에 비해 너무 작아서, 부호가 완전히 맞아도 두 박스
footprint의 약 59%가 겹칠 수밖에 없었다(기하학적으로 당연한 결과 - 버그 아님).
남아있던 크기 오차(측정 5.2~6.3cm vs 참값 3.6cm, 절대오차 1.5~2.5cm)도 참값이
작다 보니 상대오차로는 40~70%나 되어 눈에 띄게 부정확해 보였다.

**조치**: 오프셋을 물리적 안정 범위 안에서(Large 반폭 0.115/0.08 대비 절반
이하) 7.2cm로 넓혔다(`35.crate_scan_setup.py`의 `TABLE_BOXES`에서 Small의
오프셋을 (0.03,0.08)->(0.06,0.14)로 변경, Large는 그대로). 오프셋의 절대오차는
알고리즘의 한계로 여전히 남아있지만(같은 정도의 point cloud 노이즈이므로),
참값이 커지면 상대오차는 자동으로 줄어든다.

Isaac Sim을 재실행해 새 point cloud(9시점, 724,732점)를 얻고 5회 반복 검증:

| 항목 | 결과 |
|---|---|
| 검출된 박스 개수 | 3/3 (5회 모두) |
| 오프셋 방향 | 5/5 모두 두 축 다 부호 일치 |
| 오프셋 크기(측정 평균) | base_link (0.0339, -0.0560)m |
| 오프셋 크기(참값, world 90도 회전 적용) | base_link (0.04, -0.06)m |
| 오프셋 절대오차 | 7.3mm (참값 대비 상대오차 ~10%, 이전 40~70%에서 크게 개선) |
| 반복 간 표준편차 | X: 0.7mm, Y: 1.3mm (매우 안정적) |

raw point cloud를 직접 시각화해서(높이=색) 재구성된 Large 사각형이 실제 노출된
teal색 영역과 잘 겹치는 것도 육안으로 재확인했다.

**최종 결론**: 동일 크기 적층 박스 검출은 개수/존재/높이/footprint 크기/XY
오프셋(방향+크기 모두) 전 항목에서 반복 가능하고 정확하다. 원래 3.6cm의 극히
작은 오프셋 케이스는 "정보이론적 한계에 가까운 어려운 케이스"로 여전히 유효한
관찰이지만(0절 참고), 이 데모의 최종 시나리오는 7.2cm 오프셋으로 확정한다.

### 3-10. 유령 4번째 박스 (RANSAC 조각을 독립 박스로 오인)

실제 데이터로 반복 실행하던 중, 위 모든 수정을 적용한 뒤에도 매번 폭 4~5cm의
가늘고 긴 4번째 "박스"가 결과에 남았다(fill_ratio 0.70~0.75, 위치는 Small/Large
스택과 무관한 곳). 이 데모의 실제 박스 중 가장 좁은 변도 0.12~0.13m(Medium)라서,
4~5cm짜리 footprint는 물리적으로 실제 박스일 수 없다 - RANSAC이 우연히 만들어낸
가늘고 긴 조각으로 판단(`box_geometry.MIN_BOX_SIDE_M=0.04`를 근소하게 통과).

**수정**: `box_geometry.py`의 전역 하한(다른 legacy 단일 시점 스크립트에 영향을
